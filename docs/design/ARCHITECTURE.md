# 哲思屋（Philosophia）— 辩证法辩论系统

> 用古希腊的辩论形式，通过多方哲学视角的交锋，逼近真理

**状态**：设计文档 v0.1
**创建时间**：2026-04-29
**作者**：Liber1917

---

## 一、项目愿景

### 核心理念

哲思屋是一个**通用化的多智能体辩证法辩论系统**，它不是为辩论而辩论，而是通过辩论**逼近真理**。

这对应古希腊辩证法（διαλεκτική）的原初含义——通过问答对话揭露矛盾，最终在更高层次达成新的认识。

### 设计原则

| 原则 | 说明 |
|------|------|
| **真理优先** | 辩论是通向真理的手段，轮次、胜负都是次要的 |
| **证据驱动** | 论证必须基于可查证的事实，而非空洞的修辞 |
| **证据可共享** | 各方搜索到的证据汇聚到共享证据池 |
| **悬置存疑** | 无法判定的论点不强行定论，暂时悬置 |
| **书记记录** | 辩论过程被完整记录，形成可查阅的知识库 |

## 三、核心组件

### 3.1 编排层（Orchestrator）

**职责**：
- 注入 `search()` 工具给各哲人Agent
- 管理并行检索调度（多个Agent可同时搜索）
- 控制辩论流程：正题 → 反题 → 矛盾展开 → 综合
- 维护辩论状态：当前轮次、活跃参与者、未解决的争议点

**设计原则**：
- 不修改哲人Skill的内部逻辑
- 通过工具注入实现能力扩展
- 支持动态增减参与者

### 3.2 书记Agent（Scribe）

**职责**：
- 实时记录各方论点，标注来源
- 管理共享证据池，对每条证据进行可信度评估
- 维护"悬置区"（Epoché），记录存疑论点
- 识别收敛信号，判断辩论是否达到新认识
- 辩论结束时生成综合报告并持久化

**哲学渊源**：
古希腊辩论（Symposion）有专职书记记录，苏格拉底的对话录也是对话的记录。书记不是裁判（评判胜负），而是记录者和综合者。

### 3.3 哲人Agent（Philosopher）

每个哲人是一个独立的Skill，通过编排层注入搜索能力。

**热插拔机制**：哲人通过 `philosopher-registry.json` 声明式注册，运行时从远端仓库动态加载。

**标准接口**：
```
输入：
- 辩题
- 自己在辩论中的角色（正题/反题/追问）
- 历史辩论上下文（各方论点、证据）
- 可用工具：search()

输出：
- 本轮论点
- 引用的证据列表
- 对对方论点的回应
- 主动识别需要搜索的内容
```

### 3.4 证据持久化层

辩论过程中的所有内容持久化到工作区：

```

├── archive/                    # 历史辩论存档
```

## 四、证据系统

### 4.1 证据生命周期

```
证据生成 → 提交证据池 → 交叉检验 → 可信度标注 → 被引用/悬置
                              ↓
                         ┌────┴────┐
                         ↓         ↓
                      [通过]    [存疑]
                         ↓         ↓
                    ⭐⭐⭐⭐⭐   进入Epoché
```

### 4.2 证据数据结构

```json
{
  "id": "ev-{timestamp}-{hash}",
  "content": "具体事实陈述",
  "source": {
    "name": "来源名称",
    "url": "链接",
    "type": "论文/报告/新闻/网页"
  },
  "searched_by": "哲人agent标识",
  "search_time": "ISO时间戳",
  "credibility": {
    "cross_verified": true,
    "sources_agree": true,
    "stars": 5,
    "notes": "可信度说明"
  },
  "used_by": ["引用此证据的agent列表"],
  "epoché": false
}
```

### 4.3 交叉检验机制

**检验流程**：
1. **来源核查**：来源是否权威？是否有利益相关？
2. **多源比对**：是否有其他独立来源印证？
3. **逻辑自洽**：数据与论点推论是否一致？

**可信度等级**：
| 等级 | 标识 | 说明 |
|------|------|------|
| 5星 | ⭐⭐⭐⭐⭐ | 多源印证，来源权威，逻辑自洽 |
| 4星 | ⭐⭐⭐⭐ | 单源权威，基本可信 |
| 3星 | ⭐⭐⭐ | 需进一步验证，暂可引用 |
| 2星 | ⭐⭐ | 来源存疑，谨慎引用 |
| 1星 | ⭐ | 可信度低，仅作参考 |

### 4.4 悬置（Epoché）机制

**触发条件**：
- 证据来源相互矛盾，无法判定
- 论点涉及形而上学问题，无法用经验验证
- 证据不足，需要等待更多信息

**悬置项结构**：
```json
{
  "id": "ep-{timestamp}-{hash}",
  "claim": "无法判定的论点",
  "submitted_by": "提出此论点的agent",
  "epoché_reason": "悬置原因",
  "pending_evidence": ["需要进一步查证的内容"],
  "created_at": "ISO时间戳"
}
```

**悬置原则**：
> 悬置不是否定，而是暂时搁置判断，保持认识的开放性。这与胡塞尔现象学的"悬置"（Epoché）概念一致——将判断悬搁，以便让事物本身显现。

## 五、辩论流程

### 5.1 完整辩论流程

```
┌─────────────────────────────────────────────────────────────┐
│                      阶段一：开题                            │
│  用户输入辩题                                                │
│  编排层解析辩题，识别核心概念和争议点                        │
│  选择参与辩论的哲人（基于辩题匹配）                           │
│  分配角色：正题方 / 反题方 / 追问方                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      阶段二：正题                            │
│  正题方哲人陈述观点                                          │
│  主动搜索支持证据                                            │
│  书记记录论点和证据                                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      阶段三：反题                            │
│  反题方哲人提出反驳或对立观点                                 │
│  主动搜索反驳证据                                            │
│  可能对正题方的证据进行交叉检验                               │
│  书记更新证据池，识别存疑项                                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      阶段四：矛盾展开（循环）                 │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 追问方A：揭示正题与反题之间的深层矛盾               │   │
│  │ 正题方回应                                         │   │
│  │ 书记记录张力所在                                   │   │
│  └─────────────────────────────────────────────────────┘   │
│  （可多轮，取决于收敛信号）                                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      阶段五：综合                            │
│  书记识别收敛信号                                            │
│  提炼"新认识"（不是妥协，而是更高层次的认识）                │
│  生成综合报告                                                │
│  用户决定是否继续追问或结束                                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      阶段六：持久化                          │
│  保存辩论记录到 dialectike-records/                          │
│  生成综合报告                                                │
│  更新悬置区记录                                              │
│  可选：分享辩论记录                                          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 收敛信号

辩论应继续的条件：
- 各方论点仍在交锋，未出现一方完全认输
- 仍有新的证据或视角涌现
- 悬置区有待解决的存疑项
- 用户选择继续追问

辩论可收敛的条件：
- 某方开始部分承认对方论点（趋向共识）
- 出现了更深层次的问题（当前层面讨论已充分）
- 某方主动说"我不确定"（触及认知边界）
- 重复论点开始出现（讨论已饱和）
- 用户判定可以结束

### 5.3 用户交互

```
辩论过程中用户可：

[追问]  指定某个哲人回答特定问题
[质疑]  对某个论点提出质疑，触发新一轮搜索
[悬置]  将某个论点标记为存疑
[保存]  随时保存当前辩论状态
[中断]  终止辩论，保存记录
[继续]  在已有辩论基础上继续
```

## 七、技术实现

### 7.1 技术栈

| 组件 | 技术选择 |
|------|---------|
| 核心框架 | WorkBuddy Agent Framework |
| 哲人Skill | 女娲(Nuwa)方法论蒸馏的Skill（远端热插拔） |
| 搜索能力 | Web Search（编排层注入） |
| 持久化 | Markdown + JSON（文件系统） |
| 版本控制 | Git + GitHub |

### 7.2 热插拔架构

```
哲人注册中心（philosopher-registry.json）
    │
    ├── hegel: { source: "remote", url: "https://raw.githubusercontent.com/..." }
    ├── marx:  { source: "remote", url: "https://raw.githubusercontent.com/..." }
    └── ...

编排层（dialectike SKILL.md）
    │
    ├── 读取 registry.json
    ├── 过滤 enabled: true
    └── web_fetch(url) → 动态加载 SKILL.md
```

### 7.3 Skill体系

```
skills/
├── dialectike/              # 辩证法编排核心Skill
│   ├── SKILL.md            # 主编排逻辑（支持热插拔）
│   └── prompts/            # 提示词模板
│
philosophia-debate（主仓库）
│
└── philosopher-registry.json  # 哲人注册中心（声明式配置）
    │
    └── 各哲人Skill通过远端仓库热插拔
        ├── hegel-perspective-skill（独立GitHub仓库）
        ├── karl-marx-skill（独立GitHub仓库）
        └── ...（未来更多哲人）
```

### 7.4 远端热插拔流程

```
1. 启动 dialectike
   ↓
2. 读取 philosopher-registry.json
   ↓
3. 遍历所有 enabled: true 的哲人
   ↓
4. 对 source: "remote" 的哲人：
   - web_fetch(url) 获取 SKILL.md 内容
   - 注入到上下文中
   ↓
5. 辩论开始

添加新哲人：
1. 创建独立的 GitHub 仓库（如 xxx-perspective-skill）
2. 在 philosopher-registry.json 中添加条目
3. 无需任何脚本，立即生效
```

### 7.3 关键接口

**编排层 → 哲人Agent**：
```python
# 注入搜索能力
{
  "tool": "search",
  "description": "搜索支持辩论论点的证据",
  "params": {
    "query": "搜索关键词",
    "scope": "academic|news|general"
  }
}

# 调用哲人Agent
{
  "philosopher": "hegel",
  "role": "thesis",  # thesis | antithesis | critique
  "context": {
    "topic": "辩题",
    "previous_arguments": [...],
    "evidence_pool": [...],
    "epoché_items": [...]
  }
}
```

**哲人Agent → 编排层**：
```python
{
  "argument": "本轮论点内容",
  "evidence_requests": [
    {"query": "搜索需求1", "purpose": "支持论点"},
    {"query": "搜索需求2", "purpose": "反驳对方"}
  ],
  "references": ["evidence-id-1", "evidence-id-2"],
  "responses_to": ["arg-id-1", "arg-id-2"],
  "epoché_suggestions": ["建议悬置的论点"]
}
```
